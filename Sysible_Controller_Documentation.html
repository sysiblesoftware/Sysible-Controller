Section 1Introduction to Sysible Controller

Sysible Controller is the central management console of Sysible Enterprise Software. It is installed on a single Linux machine — the controller — and from there gives a system administrator one place to see, configure, and act on an entire fleet of Linux hosts. The controller is made up of two cooperating pieces: a FastAPI backend service that holds the fleet's inventory, credentials, and task queue in a local SQLite database, and a PySide6 desktop GUI client that the administrator runs to interact with that backend. Both pieces are installed together and run on the same machine by default, with the desktop client talking to the backend over HTTPS exactly as any other client of the API would.

Sysible Controller manages target hosts through two independent mechanisms, and an administrator can mix both across a fleet:

  • The Sysible agent — a small Python daemon installed on each managed host as its own systemd service, where it heartbeats back to the controller, polls for queued tasks, and reports results.
  • Direct SSH — hosts that cannot or should not run a persistent agent are instead "enrolled" by handing the controller SSH credentials once. The controller then pushes its own auto-generated SSH key to the host and drives it directly over SSH, including a live interactive terminal session from the GUI.

Both mechanisms feed the same fleet-wide views in the GUI — User & Group Administration, Service Management, System Health & Logs, Cron & Systemd Timers, and so on — so an administrator generally does not need to think about which transport a given host uses once it is enrolled.

Beyond the admin-facing GUI, the controller also runs an optional, separate host-facing Webserver Portal as its own process. This is what an end user or host operator — not the Sysible administrator — visits in a browser to self-service: downloading a ready-to-run agent bundle for their machine, or grabbing files the administrator has staged for them, without ever needing admin credentials or shell access to the controller itself.

Security model

HTTPS everywhere Admin API key Per-host SSH keys One-time enroll tokens Role-based access Runs as you, not root Login throttling

Security is layered throughout the product:

  • Every network-facing service — the backend API, the Webserver Portal, and the agent's calls home — is HTTPS-only, using a self-signed TLS certificate the installer generates specifically for that machine.
  • Every call the desktop GUI makes to the backend is gated by a single admin API key issued at install time and sent as an X-API-Key header.
  • Each administrator login issues a signed, expiring per-administrator token. The identity an action runs as on a host is derived from that token server-side, so it cannot be forged or chosen by the client.
  • Every action on a host runs as the administrator's matching local user (via runuser), with that user's own sudo rights and the host's own audit trail — not as an anonymous root daemon. Privileged steps are attempted unprivileged first and escalated only when the OS reports a privilege error.
  • Role-based access: administrators are Superusers or Sysadmins. Managing administrators and viewing the Live Activity & Controller Log are restricted to Superusers, enforced on the backend rather than only hidden in the GUI.
  • Sudo passwords are never stored in the clear. For password-sudo hosts, the operator's "become" password is fed to sudo -S over stdin only — never on the command line, in the environment, in the database, or in any log. The controller holds it in memory keyed to a single task, delivers it once to the authenticated agent, and sweeps it on a short TTL. Any optional convenience copy on the operator's own workstation is encrypted at rest (Fernet, 0600 key) and namespaced per administrator.
  • Login throttling rate-limits and briefly locks out repeated failed administrator logins, and security response headers are sent on every reply with the interactive API docs disabled in production.
  • Every host managed over SSH is authenticated with a dedicated SSH key pair the controller generates and manages itself, rather than relying on shared or stored passwords.
  • The controller verifies each SSH host's key on a trust-on-first-use basis: the key is pinned on first contact and checked on every later connection (command exec, the interactive terminal, and file transfer all share one controller-side known_hosts). A changed host key — a possible man-in-the-middle, or a rebuilt machine — is refused rather than silently re-trusted, and removing a host forgets its pinned key so a legitimate rebuild can re-enroll.
  • Every agent enrollment uses a single-use enrollment token baked into its downloadable bundle, so a bundle cannot be silently reused to enroll a second, unintended host.
  • System-critical paths are guarded against destructive actions. Deleting, unmounting, or removing the /etc/fstab entry of a critical system file or mount (/, /etc, /etc/fstab, /boot, /usr, the init/systemd binaries, and so on) is blocked outright for Sysadmins and requires an explicit Superuser confirmation. The check lives in the command builders, so it holds across every front end (and even a hand-crafted API request).
  • The terminal's “Send sudo password” button is opt-in per administrator — off by default, granted only by a Superuser in Settings → Administrators, and enforced on the server (not just hidden in the UI).

Section 2System Requirements

Sysible Controller's installer (install_sysible.sh) supports Debian/Ubuntu, RHEL/CentOS/Fedora, and openSUSE/SLES families of Linux distribution. It detects which one it's running on by checking, in order, for dnf, then yum, then zypper, then apt-get, and installs the equivalent package list for whichever it finds first — it will not run on a system with none of these four available. This is the same detection order Host Software Management (Section 10) uses at runtime against managed hosts, so "which package manager does this machine use" is answered consistently whether the machine in question is the controller itself or a host it manages. The installer must be run as root; it checks this at the very start and exits immediately if it is not.

Python dependencies

Inside its own virtual environment, the controller installs FastAPI and Uvicorn (backend web framework and ASGI server), Pydantic (request/response validation), psutil (system/process metrics), paramiko (SSH client library for SSH-managed hosts), python-multipart (portal upload/login form handling), requests, and PySide6 (the desktop GUI's Qt6 bindings).

The host agent itself has a much smaller footprint — just requests — kept in its own separate requirements file so it can ship inside the downloadable agent bundle and run on a target host with no other Sysible code present.

Section 3Installation

Installation is a single script, install_sysible.sh, run with root privileges from wherever the project files currently live — it does not need to be run from its final installed location.

# from the folder containing the Sysible Controller project files
sudo ./install_sysible.sh

The script proceeds through the following stages, in order:

1. Package installation

Detects the package manager (dnf > yum > zypper > apt-get, in that order) and runs the matching install command for the full package list in Section 2 — dnf/yum install -y, zypper --non-interactive install -y (after a zypper --non-interactive refresh), or apt update followed by apt install -y.

2. Deploying project files to /opt/sysible

Sysible Controller always runs out of a single fixed location, /opt/sysible, because the CLI tooling hardcodes that path. The installer mirrors the source directory there with rsync -a --delete. A deliberate set of excludes protects live, running state from ever being wiped by a re-install: the virtual environment, the runtime run directory, logs, any *.db file, api_key.txt, hosts.json, the remote_keys directory, agent_state.json, the certs directory, and Python bytecode/cache artifacts. Re-running the installer to push a code update is therefore safe — it will not regenerate secrets, rotate the TLS certificate, or touch the database.

3. Python environment setup

Creates a virtual environment at /opt/sysible/venv, upgrades pip, and installs requirements.txt into it.

4. Making the CLI executable

chmod +x is applied to sysible_controller.

5. Generating the self-signed TLS certificate

If /opt/sysible/certs/server.crt and server.key do not already exist, the installer generates a new 2048-bit RSA key pair and a 10-year self-signed certificate. Because there is no public DNS domain to obtain a CA-signed certificate for on a LAN deployment, this self-signed cert is the controller's only option — and it is built to cover every address the controller might be reached at: DNS:localhost, IP:127.0.0.1, the machine's own hostname, and every IP address hostname -I reports, each added as its own Subject Alternative Name. This matters because Controller Configuration's "All Detected IPs (failover)" mode can hand an agent bundle any one of the controller's network interfaces, and the certificate has to cover all of them for that to verify correctly.

NoteThis step is skipped entirely if a cert/key pair already exists — so re-running the installer for a code update never rotates, and thereby breaks, a certificate already copied out to GUI machines and agents.

6. Generating the admin API key

Every admin/GUI-facing endpoint requires this key. The installer pre-generates it so the file exists with correct permissions before the service is first started; the backend will also auto-generate it on first run if it's somehow missing.

7. Installing the backend as a systemd service

Writes /etc/systemd/system/sysible-backend.service, running uvicorn backend.app:app out of /opt/sysible, bound to 0.0.0.0:9000, using the certificate just generated, restarting automatically on failure. The service is enabled (starts on boot) but deliberately not started by the installer — that happens via sysible_controller start afterward.

8. Installing the CLI

Copies sysible_controller to /usr/local/bin/sysible_controller, chmod +x'd and on the system PATH.

9. Installing the application menu launcher

On a machine with a desktop environment, adds a Sysible Controller entry to the system's application menu, using the same logo shown throughout the GUI. Clicking it prompts graphically for the admin/root password (via pkexec) and runs sysible_controller gui — the same one-click way back into the dashboard described in Section 5, with no terminal required. This step only runs if both pkexec and /usr/share/applications are present, so it's a silent no-op on a headless server with no desktop at all.

Finishing upThe installer's final output tells you exactly where the admin API key lives (/opt/sysible/api_key.txt, mode 600, root-only) and where the TLS certificate is (/opt/sysible/certs/server.crt). Copy that certificate file — never the .key file — to any machine that will run the GUI client (if not this same machine) and to every host that will run the agent, pointing each at it via SYSIBLE_CA_CERT so they can verify the controller instead of blindly trusting it. The very last line of installer output is the next step:
Run: sudo sysible_controller start
Updating laterThe controller runs out of /opt/sysible, which the installer syncs from your source checkout, so a plain git pull in the checkout doesn't change what's running. To pull the latest code and redeploy in one step, run sudo sysible_controller update: it pulls the recorded checkout, re-syncs it into /opt/sysible while preserving your database, admin API key, TLS certificates, host records, SSH keys, and active sessions, refreshes dependencies, and restarts the services. (Re-running sudo ./install_sysible.sh does the same thing.) Managed hosts run their own copy of the agent, so to pick up agent-side changes you redeploy the agent to each host separately.

Section 4First Login & Administrator Accounts

Before the Dashboard or any of its tools become accessible, Sysible Controller requires every user to authenticate as an administrator. The desktop application checks the controller each time it starts and shows one of two screens — first-run account creation, or the normal login — before the Dashboard appears.

First run — create your administrator account

A brand-new installation ships with no administrator account and no default password. The first time the desktop application starts, it detects that no administrator exists yet and presents a Create Administrator Account screen instead of a login. Choose a Username and a Password (entered twice to confirm), click Create Account, and the account is created and you are logged straight into the Dashboard. There is no admin/admin default to log in with, and no separate "now change your password" step afterward — you pick your own password up front. The password must satisfy the Administrator Password Policy in effect (see Section 9). This screen only appears while no administrator exists; once the first account is created it never appears again.

Logging in

Every launch after the first shows the normal Administrator Login screen, which displays the Sysible logo and asks for the Username and Password of an existing administrator. If credentials are incorrect, an error message indicates an invalid username or password and the password field is cleared. If the backend cannot be reached at all, a different error reports that connection problem instead.

Forced password change for temporary passwords

When one administrator adds another from Sysible Controller Settings, the new account is created with a temporary password and flagged to change it. The first time someone logs in on such an account, a Change Your Password dialog appears automatically and cannot be dismissed — by design, an administrator cannot continue into the Dashboard while still using a temporary password set by someone else. The new password must satisfy the Administrator Password Policy currently configured (see Section 9). Once set, future logins on that account use the new password directly, unless a fellow administrator later resets it back to a temporary one, in which case the same flow runs again. The first-run account you create for yourself is never subject to this, since you chose its password from the start.

NoteThis is a one-shot login check, not a session/token scheme — the GUI is a single long-lived desktop process used by one person at a time. The Webserver Portal (Section 11), used by potentially many concurrent host operators in a browser, does use real session tokens. Additional administrator accounts are managed from Sysible Controller Settings once logged in, and every login attempt and account change is recorded to the audit log.

Section 5The sysible_controller Command Line

sysible_controller is the single command-line entry point for operating the controller once installed. It wraps systemctl for the backend's systemd service (sysible-backend) and separately manages the desktop GUI client as an ordinary background process — the GUI is one administrator's desktop application, not a daemon, so it is deliberately not made into a systemd service itself.

sysible_controller {start|stop|restart|status|logs|gui|destroy}
CommandRoot?What it does
startYesStarts sysible-backend, then polls https://127.0.0.1:9000/ until the API is actually reachable (not just "systemd says active") and confirms /openapi.json advertises current routes. Refuses to proceed if the TLS certificate is missing. Then launches the desktop GUI as a background process.
stopYesStops the GUI process (if tracked), stops sysible-backend, kills any leftover Webserver Portal subprocess, and as a last resort uses lsof to kill anything still bound to port 9000.
restartYesExactly stop followed by start.
statusNoPrints systemctl status sysible-backend plus whether a GUI process is currently running (and its PID).
logsNoRuns journalctl -u sysible-backend -f to tail the backend's live log stream.
guiYesStarts only the desktop GUI, without touching the backend. Use this to reopen the dashboard after it's been closed. Works over an SSH X11 session too — see Remote GUI access over SSH below.
enable-remote-guiYesOpt-in: turns on SSH X11 forwarding (installs xauth and the Qt libxcb-cursor0 library, sets X11Forwarding yes in sshd_config, reloads sshd) so an administrator can run the desktop GUI over SSH/PuTTY. A base install never alters sshd.
disable-remote-guiYesReverts the above (X11Forwarding no, reloads sshd).
destroyYesIrreversible uninstaller — see warning below.
The dashboard window doesn't quit on closeClosing the dashboard's window minimizes it to a system tray icon rather than quitting the application (falling back to an actual close only on desktops without tray support). The normal way to bring it back is the tray icon's "Open" action; sysible_controller gui is the command-line equivalent, useful if the tray icon was dismissed or the GUI process was killed outright — and on a machine with a desktop environment, the Sysible Controller application menu launcher installed in Section 3, Step 9 does the same thing with a click instead of a terminal, prompting graphically for the admin/root password it needs.
destroy is irreversibleIt prints an explicit warning and requires typing the literal word destroy to confirm — anything else aborts with no changes made. Once confirmed it: stops everything, disables and deletes the sysible-backend systemd unit, takes a timestamped backup of the SQLite database to /tmp/sysible-backup-<timestamp>.db (left behind for you to delete once no longer needed), deletes the entire /opt/sysible directory — database, TLS certificate/key, and admin API key all gone — and removes its own /usr/local/bin/sysible_controller binary. destroy never reaches out to already-enrolled hosts: their agent services keep running locally and will simply start failing to reach a controller that no longer exists. Disenroll hosts first (Section 7), or run disenroll_agent.sh on each one, if you want them cleaned up too.

Remote GUI access over SSH (X11)

The desktop GUI is a Linux application, but an administrator does not have to install it on their own machine to use it. Running sudo sysible_controller enable-remote-gui configures the controller for SSH X11 forwarding, so an administrator can connect over SSH and run sysible_controller gui on the controller while its window appears on their screen — the GUI runs where it already lives, only the display is forwarded. This is how a Windows administrator can use the full native GUI with no Windows install of the client at all.

The administrator's own machine needs an X server (built in on Linux; XQuartz on macOS; VcXsrv or X410 — or the all-in-one MobaXterm — on Windows) and an SSH client with X11 forwarding enabled (ssh -Y, or, in PuTTY, Connection → SSH → X11 → Enable X11 forwarding). With those in place, sysible_controller gui opens the dashboard on their desktop.

Run it as your own user, not with sudoThe GUI must run as the connecting user so it can reach that user's forwarded display — sudo runs it as root, which has no X authorisation cookie, so the window never appears. sysible_controller gui prompts once for your password only to read the admin API key; the GUI itself runs unprivileged on your forwarded display. Verify forwarding first with echo $DISPLAY (it should read localhost:10.0 — not empty, and not :0, which is the controller's own console) and a quick xeyes. Turn the feature back off with sudo sysible_controller disable-remote-gui.

Section 6The Dashboard

After logging in, you land on the main Dashboard — the home screen for the whole application. It is a single window with the Sysible logo and product name at the top, a search bar, and a grid of five clickable tiles below it. Each tile opens that tool in its own window ("popout"), which can be left open alongside the Dashboard and other tool windows — closing a tool window does not close the Dashboard or any other open tool.

Feature search

Typing into the search bar at the top of the Dashboard matches against a curated list of common tasks — things like create a user, add a repository, restart a service, or reset password — and lists the matching task(s) below it as you type. Pressing Return, or clicking a result, opens the right tile directly; for tasks that live under System Administration's sub-tiles, it opens straight through to that sub-tile, and for User & Group Administration tasks it lands on the specific tab (Create User, Account, Password, Groups, or Reports) rather than leaving you to find it. This saves having to already know which of the Dashboard's tiles — or, for anything under System Administration, which of its eighteen sub-tiles — a given task lives under.

Light and dark theme

A small pill-shaped switch in the top-right of the Dashboard header toggles the entire interface between a dark theme (the default) and a light theme. Clicking the sun or moon re-skins every window already on screen — the Dashboard and any open tool popouts — immediately, with no restart, and the choice is remembered the next time you launch the GUI. The toggle deliberately uses plain sun/moon glyphs rather than an icon font, so it always stays usable as the way back out of light mode even if the icon font is unavailable.

Signed-in identity & logging out

The Dashboard header shows Signed in as <user> with the current administrator's role, next to a Log Out button (also available on the system-tray menu). Logging out revokes this session's login token on the controller, closes the Dashboard and every open tool popout, and returns to the login screen — without stopping the backend, so the same or another administrator can sign straight back in. It's distinct from Quit Sysible Controller (the tray menu), which ends the GUI process, and from sysible_controller stop, which stops the backend service as well.

Your sudo password

Next to the Log Out button, a Sudo Password button opens a small dialog where any administrator — Superuser or Sysadmin — stores or clears their own sudo ("become") password. It's the credential the controller delivers to the agent to elevate on password-sudo hosts, and the same password the terminal's Send sudo password button uses. It lives here, on the dashboard, rather than in Settings, precisely so a Sysadmin (who doesn't see Settings) can set it.

  • Scope — store one password as the fleet default (used on every password-sudo host), or pick a specific host for a per-host override.
  • At rest — encrypted on your own workstation (Fernet, 0600 key) and namespaced to your account, so one administrator's password is never used as another's. Nothing is stored on the controller.
  • Status & clear — the dialog shows whether a password is currently stored for the selected scope, and lets you clear it.

Live Activity & Logs (Superusers only)

The Live Activity & Logs tile opens a two-tab window reserved for administrators with the Superuser role — the people who run the controller itself, as opposed to the Sysadmins who run the fleet. The restriction is enforced on the backend, so the underlying activity and controller-log endpoints reject a non-Superuser token even if the tile were somehow reached directly.

  • Activity — a chronological feed of operator actions across the fleet: who did it, on which host(s), and a short human-readable summary (for example Create user 'jdoe' on 2 hosts: web-01, db-01, Locked 'test2' on dev1, Ran ad-hoc command/script on 3 hosts: …). The same action dispatched to several hosts is collapsed into a single row — rather than one near-identical row per host — with the hosts summarized by environment (for example all prod servers, or all servers (all environments) when an action spans the whole fleet; a partial set falls back to the host names), by grouping entries with the same operator, summary, and command within a short window. Routine background bookkeeping — the periodic user-sync and SSH-enable probes — is deliberately excluded so the feed reflects real operator intent.
  • Controller Log — a live tail of the backend service's own log, for diagnosing the controller itself.

Double-click any activity entry to pop out an Activity Detail window showing who/what/where/when and the complete command or script that ran (the feed row only shows a one-line summary, since a script can be many lines), with a Copy command button. Both tabs auto-refresh, with a manual Refresh now as well, and the activity feed can be cleared from the view.

Section 7Host Enrollment

The Host Enrollment page is where hosts are brought under management using the Sysible agent, and where the resulting fleet of enrolled hosts is organized and maintained. It is reached from the "Sysible Controller Host Enrollment" tile on the Dashboard.

Community edition — 10-host limitThis build is the Community edition, which manages up to 10 hosts. The controller refuses to enroll an 11th host (agent or SSH), and this page shows a Community edition — N/10 hosts used badge so the limit is visible up front rather than only surfacing as an error. The cap is defined in one place (backend/edition.py); larger-fleet and unrestricted use are intended for a separately licensed Enterprise edition.

Downloading and running the agent bundle

Clicking Download Agent Bundle prompts you to choose where to save sysible-agent-bundle.zip — the same ready-to-run bundle the Webserver Portal hands to end users, built here on demand. Each download bakes in a fresh, one-time enrollment token plus the controller's configured address and port. The zip contains:

  • agent.py — the Sysible host agent program itself
  • requirements.txt — the Python packages the agent needs (installed automatically)
  • sysible_agent.env — the controller's address and this download's one-time enrollment token
  • server.crt (when applicable) — the controller's TLS certificate, so the agent can verify it's talking to the real controller
  • a systemd service unit file for the agent
  • run_agent.sh — the installer script (run as root on the target host)
  • disenroll_agent.sh — a companion script that cleanly removes the agent from that host

To enroll a host: copy the unzipped bundle onto the target machine, make run_agent.sh executable, and run it as root:

# on the target host
chmod +x run_agent.sh
sudo ./run_agent.sh
Optional: run the agent unprivilegedBy default the agent service runs as root. To run it as a non-root account with passwordless sudo instead, install with sudo ./run_agent.sh --unprivileged (creates a dedicated, locked sysible system account) or sudo ./run_agent.sh --user=NAME (an existing account you manage). The installer still needs root — it creates the account, writes an /etc/sudoers.d/sysible-agent drop-in (validated with visudo) granting passwordless sudo, and sets User= on the unit; the agent then escalates each command via sudo -n. Because the toolset spans the whole system, the account needs full NOPASSWD sudo, so the benefit is no root login and a sudo audit trail rather than a smaller blast radius. disenroll_agent.sh removes the sudoers drop-in and the sysible account if it created one.

The agent installs and starts as a background systemd service that restarts itself on crash or reboot. The host then appears in the Enrolled Hosts list (allow a few moments for the first heartbeat).

One bundle, one hostEach bundle's enrollment token is single-use — download a fresh bundle per host rather than reusing one zip on a second machine.

Command-line bundle download (curl)

For a terminal-only or headless host that can't open a browser, the Command-Line Bundle Download (curl) box on this page gives a one-liner that downloads the agent bundle, unzips it, and runs the installer in a single shot — authenticating with the Webserver Portal's login over HTTP Basic auth (curl -u). Click Copy to Clipboard, replace <password> with the real portal password, and paste it on the target host:

curl -k -sS -f -u 'ops:<password>' -o sysible-agent-bundle.zip \
  "https://CONTROLLER:PORT/cli/bundle" \
  && unzip -o sysible-agent-bundle.zip -d sysible-agent-bundle \
  && cd sysible-agent-bundle && chmod +x run_agent.sh && sudo ./run_agent.sh

It needs the Webserver Portal running with login credentials set (configured on the Webserver Portal page); the command is pre-filled here with the controller's address, the portal port, and the portal username. -k skips the self-signed-certificate check and the final install step needs sudo.

Environment tagging

Hosts can be organized into environments — labels such as "Production," "Staging," or "Development." Environments are entirely optional; an unassigned host is listed under an "Unassigned" heading. Select a host, use the Environment dropdown (choose "+ New environment..." to create one on the fly), and click Set Environment to apply it. The same environment list is shared elsewhere in the application, including Sysible Connect's SSH-enrollment flow. To delete an environment, select it in the dropdown and click Remove Environment; it must have no hosts still assigned (reassign them first), so removing an environment never orphans a host.

Bulk assignment. You don't have to set environments one host at a time. Sudo policy — passwordless or password ("become")

Because every action runs as the triggering administrator's matching local user (see the security model), how that user obtains root matters. The Sudo policy box sets this per host (and per environment):

  • NOPASSWD — the user has passwordless sudo on that host; privileged steps just work.
  • Password required ("become") — that host requires a sudo password. Modeled on Ansible's become-password; the controller delivers your stored password to the agent over stdin for just that task. See Sysible Connect and the security model for how the password is handled.

Use Requires Password / Passwordless (NOPASSWD) on the selected host(s), and Set Environment's Sudo Default… so every host assigned to that environment inherits the setting automatically (applied at assignment time; an individual host can still be overridden afterward).

Your sudo password isn't set hereThe host's sudo policy lives on this page, but your personal sudo password is a per-administrator credential — set it from the Sudo Password button in the dashboard header (see Your sudo password), where any administrator (Superuser or Sysadmin) can store or clear it. A dispatched action on a password-sudo host with no stored password stops with a clear message pointing you there, rather than failing silently.

Preventing duplicate enrollment

The controller refuses to enroll a machine whose IP is already managed — by an agent or another SSH record — so the same physical box can't end up as two separate hosts under different names. An agent's automatic SSH side, enrolled when it first accepts the controller key, likewise absorbs any earlier manual SSH record at the same IP. And throughout the GUI the host lists de-duplicate by IP, so a host reached both ways shows as one row.

This only requires an SSH server to be running on the host; if none is, the host stays agent-only and the page explains that an SSH server would be needed for a live terminal.

The pop-out terminal

Double-click a host to open its terminal in its own window. The terminal is a pop-out (not docked in the page), so you can place it wherever you like, resize it freely, and keep several open side by side. You can open more than one session to the same host at once — each double-click opens a new, independent session, and repeated windows for the same host are numbered in their title bars.

For a host that's enrolled both ways, a Connection picker at the top of the window switches between the SSH session and the Agent console; single-transport hosts show only what they have.

The terminal opens as your own administrator user on the host (the local account matching the administrator you logged in as), not as root — the same run-as-you model the fleet tools use — so your shell history, environment, and sudo rights are your own.

On an SSH connection the terminal is a genuine PTY-backed interactive session. Full-screen, cursor-addressed programs — vim, top, htop, less — render correctly rather than spilling stray redraw lines, and the terminal resizes with the window so those programs use the whole panel. Output is colorized (ANSI colors are honored), and the user@host portion of the prompt is shown in green, turning red whenever you are root (including after su or sudo -i) as an at-a-glance privilege cue. Arrow keys, Tab, and Backspace all work, and pasting sends text straight into the session. Buttons provide Send Ctrl+C (a real interrupt signal), Clear (clears the pane and redraws the prompt), and Exit Session (closes the window and tears down the remote session).

Each terminal window also has a toolbar with Upload… and Download… (transfer a file to or from that host over SSH), Find… (search the terminal output), Save Output… (write the terminal's text to a file), and A- / A+ font-size controls (the remote pty is re-sized to match so full-screen apps still fill the panel).

On an Agent connection there's no persistent shell, so true interactive sessions aren't possible. Instead, type a command in the bottom box and press Enter — it's queued for the agent to execute, with output, errors, and exit code returned once it reports back (typically a few seconds; a timeout is reported after about 30 seconds of silence). Command history is recalled with the Up/Down arrow keys.

File upload and download

The File Transfer section moves files to and from a managed host. SSH hosts transfer over SFTP with no enforced size limit, completing synchronously. Agent-enrolled hosts transfer through the same queued-task channel used for commands, capped at a modest size (the page states the current limit) since that channel isn't designed for bulk data movement.

The target host must run an RDP serverRDP connects to a remote-desktop server listening on the host (TCP 3389) — it isn't provided by the Sysible agent. You're prompted for the host, username, and password, with a resolution option (the default fits the session to the window). Section 12Common Tasks

The sections above explain what each tool is and why it works the way it does. This section is the opposite: short, numbered recipes for every button, field, and dialog in Sysible Controller, organized by the tool you'd find them in. If you already know which action you want and just need the exact clicks, start here — for the reasoning behind a behavior, follow the cross-reference back to its section above.

Logging in

Log in as an administrator

  1. Start Sysible Controller (sysible_controller start, or sysible_controller gui if the backend is already running) — see Section 5.
  2. In the Sysible Controller Administrator Login dialog, enter your Username and Password. The default seeded account is admin / admin until changed or another administrator is added.
  3. Click Log In, or press Enter from the Password field.
  4. An invalid combination shows "Invalid username or password," clears the Password field, and refocuses it. An unreachable backend shows a connection error instead.
  5. Click Quit at any time to exit without logging in.

Set a new password when prompted

Shown immediately after a login whose account has "must change password" set — a freshly seeded admin/admin account, or one another administrator just (re)created with a temporary password. There is no Cancel; the dashboard does not appear until a new password is set.

  1. Enter New Password and Confirm Password.
  2. Click Set Password, or press Enter from the Confirm field.
  3. An empty password or a mismatch between the two fields is rejected in place (the Confirm field is cleared and refocused on a mismatch).
  4. On success the dialog closes and the dashboard appears.

Reopen the dashboard without a terminal

If the dashboard window was closed (or "Quit Sysible Controller" was chosen from its tray icon) while the backend keeps running, there's no need to SSH in or open a terminal to get back in.

  1. Open the host's application menu and click the Sysible Controller icon (installed by Section 3, Step 9 — skipped if no desktop environment was detected at install time).
  2. Enter the admin/root password at the graphical prompt this triggers.
  3. The dashboard reopens and the Sysible Controller Administrator Login dialog appears, exactly as if sysible_controller gui had been run from a terminal.

Sysible Connect

Dashboard tile: Sysible Connect. See Section 8 for the terminal and file-transfer model.

Connect to a new host over SSH

  1. In the SSH to a New Host (Not Yet Joined) section, fill in Host name, IP address, Username (defaults to root), and the host's current SSH password.
  2. Choose an Environment, or + New environment... to create one.
  3. Click Connect Host. The controller installs its own SSH key on the host using the password you gave it once, then discards that password — every connection afterward is passwordless.
  4. The host now appears in Managed Hosts.

Open a terminal session on a managed host

  1. Select a host in Managed Hosts.
  2. SSH hosts open a live, fully interactive terminal — sudo prompts, vim, arrow keys, and Ctrl+C (via the dedicated Send Ctrl+C button, which only appears during a live SSH session) all work as they would in a normal terminal.
  3. Agent hosts use a type-and-submit model instead: type a command and send it, wait while it's queued and polled (up to 30 seconds), then read the result once it returns. There's no partial output and no way to interrupt a running command.
  4. Click Clear to clear the terminal pane's contents.

Upload a file to a host

  1. Click Browse... and choose a local file.
  2. Enter the destination path on the remote host.
  3. Click Upload.
  4. SSH hosts transfer synchronously over SFTP with no size limit. Agent hosts queue the transfer through the same task channel as commands, capped at the agent file-transfer size limit, and it's polled until done.

Download a file from a host

  1. Enter the source path on the remote host.
  2. Click Download... and choose where to save it locally.
  3. Same SSH-vs-agent behavior as uploading: immediate SFTP for SSH, capped and polled for agent hosts.

  1. Select it in Managed Hosts, click Remove Host.

  1. Click Show Sysible Controller Public Key.

  4. Click Connect. Sysible launches FreeRDP (xfreerdp), or Remmina if that's what's installed.

CommandRoot?Summary
